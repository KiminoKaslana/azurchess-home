**说明**

- 所有 HTTP 接口均为 `POST`。  
- 路由取 URL 第一段路径：例如 `http://user-server/Login` 中的 `Login`。  
- 权限控制依赖请求头里的 `Token`（JWT）和用户的 `UserRole`。  
- 示例 Base URL（实际以配置为准）：  
  - 用户服 `UserServer`：`http://localhost:7000/`  
  - 游戏服 `GameServer`：`http://localhost:7001/`  

---

## 一、认证与用户管理（UserServer）

### 1. 登录：`POST /Login`

- 路径：`{UserServerBaseUrl}/Login`
- 权限：匿名
- 请求头：无特殊
- 请求体（JSON）：

```json
{
  "UserName": "testUser",
  "Password": "123456",
  "Version": "1.0.0"
}
```

- 说明：
  - `Version` 必须等于服务端当前版本（`AppConfig.Instance.Version`），否则 403。
- 响应：
  - 成功 `200 OK`：

```json
{
      "PlayerID": "655f4d8b4a3c123456789012",
      "Token": "jwt-token-string"
    }
```

    - 之后前端需要：
      - WebSocket 连接游戏服时：在请求头带上 `PlayerID`
      - 调用需要权限的 HTTP 接口时：在请求头带上 `Token`
  - 失败：
    - `400`：请求体为空
    - `403`：用户名/密码错误，或版本不匹配（文本消息为 `"版本不匹配"` / `"用户名或密码错误"`）
    - `500`：服务器异常

---

### 2. 使用 Token 登录：`POST /LoginWithToken`

- 路径：`{UserServerBaseUrl}/LoginWithToken`
- 权限：匿名
- 请求体（JSON）：

```json
{
  "Version": "1.0.0",
  "Token": "jwt-token-string"
}
```

- 逻辑：
  - 校验版本号。
  - 校验 Token 有效性、未过期、对应用户存在。
  - 若 Token 即将过期（已使用 80% 以上有效期），自动签发新 Token。
- 响应：
  - 成功 `200 OK`：

```json
{
      "PlayerID": "655f4d8b4a3c123456789012",
      "Token": "new-or-original-jwt-token"
    }
```

    - 若新 Token 被签发，则 `Token` 为新值。
  - 失败：
    - `400`：请求体为空或无 `Token`
    - `403`：Token 无效 / 已过期，响应体为错误原因文本（如 `"无效Token"`、`"登录已过期，请重新登录"`）
    - `500`：服务器异常

---

### 3. 注册：`POST /Register`

- 路径：`{UserServerBaseUrl}/Register`
- 权限：匿名
- 请求体（JSON）：直接映射到 `PlayerSaveData`，常用字段：

```json
{
  "UserName": "testUser",
  "Nickname": "测试用户",
  "Password": "123456"
}
```

- 服务器内部会：
  - 自动生成 `Id`。
  - 强制设置 `Role = Player`。
  - 设置 `LastTokenIssueTime` 为最小时间。
- 校验：
  - 同名用户：不允许。
  - 用户名格式：使用 `UserServer.ValidateName` 限制：
    - 允许：字母、数字、下划线、中文
    - 总长度 ≤ 20；中文按 2 个字符计算
- 响应：
  - 成功：`200 OK`，无响应体。
  - 失败：
    - `403`：用户名不合法（响应体 `"用户名不合法"`）
    - `412`：用户已存在（响应体 `"用户{UserName}已存在"`）
    - `400`：无请求体
    - `500`：服务器异常

---

### 4. 修改昵称：`POST /SetNickname`

- 路径：`{UserServerBaseUrl}/SetNickname`
- 权限：需要已登录玩家
- 请求头：
  - `PlayerID: <登录返回的 PlayerID>`
- 请求体（纯文本，不是 JSON）：

```plaintext
新昵称
```

- 校验：
  - 使用 `UserServer.ValidateName`，规则与用户名相同（字符集 / 长度）。
- 响应：
  - 成功：`200 OK`，无响应体。
  - 失败：
    - `400`：没有 `PlayerID`、没有请求体或昵称不合法（有错误文本 `"昵称不合法"`）
    - `500`：服务器异常（日志记录，但响应一般只给状态码）

> 建议前端：始终使用与注册时相同的规则预先校验昵称，避免连续 400。

---

### 5. 管理员修改用户角色：`POST /AdminSetUserRole`

- 路径：`{UserServerBaseUrl}/AdminSetUserRole`
- 权限：`SuperAdmin` 以上（最高管理员）
- 通用权限校验（`UserServer.Response` 统一处理）：
  - 对有 `RequiredPermission > Player` 的接口：
    - 请求头必须包含 `Token`（JWT）：
```
  Token: <jwt-token>
```
    - 后端通过 Token 查出当前用户，并验证其 `Role >= RequiredPermission`：
      - 否则：`403 Forbidden`
- 请求体（JSON）：

```json
{
  "TargetUserId": "655f4d8b4a3c123456789012",
  "NewRole": 1
}
```

- 角色枚举 `UserRole`：
  - `0`：Player
  - `1`：Admin
  - `2`：SuperAdmin
- 响应：
  - 成功：
    - `200 OK, "OK"`：角色已更新
    - `200 OK, "用户不存在或未做修改"`：目标用户不存在或角色本身无变化
  - 失败：
    - `400`：参数缺失 / `NewRole` 非法（响应 `"参数错误"` / `"角色无效"`）
    - `401`：缺少或无效 `Token`
    - `403`：当前用户权限不足（不是 SuperAdmin）
    - `500`：服务器异常

## 二、权限模型与通用约定

- 所有用户服接口都要求 `POST` 方法，否则：`405 Method Not Allowed`。
- 路由未匹配时：`404 Not Found`。
- 用户角色（`UserRole`）：
  - `Player`：普通玩家
  - `Admin`：管理员
  - `SuperAdmin`：超级管理员（可修改其他用户角色）
- JWT 相关：
  - 登录接口返回的 `Token` 需存储在前端。
  - 需要管理权限的接口，都必须在请求头带上：
```
Token: <jwt-token>
```
  - Token 有有效期，靠 `LastTokenIssueTime` + 内部过期时间控制；若提示 `"登录已过期，请重新登录"`，需要重新走 `/Login`。

## 三、管理类接口（GameServer，Admin 使用）

这些接口主要给后台或运维工具用，前端如需调用同样遵守 `Token` + `UserRole` 模型。只列出与“管理”相关的 HTTP API。

### 1. 更新伤害系数：`POST /UpdateDamageCoefficient`

- 路径：`{GameServerBaseUrl}/UpdateDamageCoefficient`
- 权限：`Admin` 及以上
- 请求头：

```
Token: <jwt-token-of-admin>
```

- 请求体（JSON示例，实际必须为5行9列）：

```json
[
  [1.0, 0.8, 1.2],
  [1.1, 1.0, 0.9]
]
```

- 说明：
  - 二维数组：`WeaponType 和 UnitType`
- 响应：
  - 成功：`200 OK, "伤害补正表更新成功"`。
  - 失败：
    - `400`：格式 / 维度错误，响应带详细错误文本。
    - `401` / `403`：Token 无效或权限不足。
    - `500`：服务器错误。

---

### 2. 更新资源信息：`POST /UpdateResourceInfo`

- 路径：`{GameServerBaseUrl}/UpdateResourceInfo`
- 权限：`Admin` 及以上
- 请求头：

```
Token: <jwt-token-of-admin>
Platform: Android      // 可选，决定写入哪个文件
```

- 请求体（JSON）：`List<ResourceInfo>`，示例：

```json
[
  {
    "Name": "AssetBundle1",
    "Hash": "abcdef1234567890",
    "URL": "https://example.com/ab/assetbundle1"
  },
  {
    "Name": "AssetBundle2",
    "Hash": "123456abcdef7890",
    "URL": "https://example.com/ab/assetbundle2"
  }
]
```

- 说明：
  - 如果有 `Platform` 头，则写入 `{Platform}ResourceInfo.json`（例如 `AndroidResourceInfo.json`），否则写 `ResourceInfo.json`。
- 响应：
  - 成功：`200 OK`，文本类似：`"成功更新 N 条资源信息"`。
  - 失败：
    - `400`：资源项缺少 `Name` / `Hash` / `URL` 等，响应为对应错误文本。
    - `401` / `403`：权限相关错误。
    - `500`：服务器内部错误。

---

### 3. 重新生成资源信息（从 OSS）：`POST /RegenerateResourceInfo`

- 路径：`{GameServerBaseUrl}/RegenerateResourceInfo`
- 权限：`Admin` 及以上
- 请求头：

```
Token: <jwt-token-of-admin>
```

- 请求体：无（空请求体即可）。
- 逻辑：
  - 需要服务端已配置并启用 OSS（`OSSManager` 不为 `null`）。
  - 接口返回成功后，会在服务端后台异步执行资源扫描和 `ResourceInfo` 生成。
- 响应：
  - `200 OK, "已计划重新生成ResourceInfo。"`：已接收任务。
  - `503, "OSS 未启用"`：OSS 功能关闭或配置不完整。
  - `401` / `403`：权限相关错误。
  - `500`：服务器异常。
